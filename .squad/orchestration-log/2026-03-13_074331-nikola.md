## Mandate

Benchmark one PBP API call (PAO vs Zalgiris game, 578 events) plus local extraction of new events since lastEventId to measure:
- Raw payload size
- Compressed (gzip) payload size
- Cold vs warm fetch latency
- Local extraction cost

## Key Results

| Metric | Value |
|--------|-------|
| Raw payload | 153.6 KB JSON |
| Gzip (on-wire) | 10.5 KB (6.8% ratio) |
| Cold fetch (TLS handshake) | ~960ms |
| Warm fetch (reused connection) | 50–66ms (avg 56ms) |
| JSON.parse only | 0.74ms |
| Full local processing (578 events) | 0.78ms |
| Filter + map from sinceId=550 (84 new) | 1.00ms |
| Total per-poll (warm) | ~57ms |

## Conclusion

Network dominates (>98% of time). Local filtering negligible. At 15s polling intervals, cost is ~2.5 MB/hour/game — negligible for a polling bot. No optimization needed at application layer unless upstream rate limits emerge.

## Decision Impact

Supports Bogdan's recommendation to defer complex payload-reduction schemes (conditional requests, quarter filters). The current implementation (client-side sinceEventId filter + gzip via Node.js default) is already optimal for this API design.
