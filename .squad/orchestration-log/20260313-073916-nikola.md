## Mission

Evaluate fastest realistic data strategy for live tracked-player event notifications. Recommend polling interval, event source (PBP vs header), and latency floor given upstream constraints.

## Critical Findings

### Latency Floor by Stage

| Stage | Latency | Controllable |
|-------|---------|--------------|
| Upstream pub (stat crew entry) | 2–15s | ❌ No |
| Poll interval wait (15s, avg) | 0–15s | ✅ Yes |
| PBP fetch (gzipped) | 0.3–0.8s | Marginal |
| Parse + match | <10ms | — |
| Throttle | 0–120s | ✅ Yes |
| Telegram send | 0.1–0.3s | — |
| **Total (typical)** | **~15–20s** | |

### PBP API Capabilities

- **Full fetch always:** `getPlayByPlay()` returns 157 KB uncompressed (~10.7 KB gzipped). No server-side filtering, ETags, or `since` params.
- **Client-side dedup works:** `sinceEventId` already implemented in adapter.
- **Alternative endpoints exist:**
  - `/api/Header`: 475 bytes gzipped (scores, clock, fouls)
  - `/api/Points`: 4.5 KB gzipped (scoring plays only)
  - Could enable "lightweight poll first" strategy if bandwidth becomes constraint

### Fastest Realistic Scenario

Reduce poll interval to **5 seconds** → saves ~7.5s average wait per cycle. At 5s intervals:
- ~7.7 MB/hour/game (gzipped)
- No observed API rate limits (tested extensively)
- Achievable latency: **avg 8–10s, worst ~21s**

**True instant (<2s) impossible without push API** — EuroLeague offers none publicly.

## Recommendation

**PBP-primary polling at 5s interval.** Make PBP the single source of truth for all events (player actions + game-level). Decouple `getLiveScore()` from the hot path; use only as fallback (30–60s cycle) for game status transitions.

**Throttling is the next bottleneck.** Default 5 msg/min will suppress roster matches. Batching (digest-style) or separate throttle tier required.
