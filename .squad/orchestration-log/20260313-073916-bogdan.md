## Mission

Design product-safe architecture for live tracked-player notifications. Balance completeness (all event types), UX (no spam), and implementation scope (Phase 1 vs later).

## Key Recommendations

### Event Classification (Configurable)

Introduce **event classes** instead of a single `NOTABLE_EVENT_TYPES` filter:

- `scoring`: made shots (2pt, 3pt, FT)
- `playmaking`: assists, rebounds
- `defensive`: steals, blocks
- `negative`: misses, turnovers, fouls
- `administrative`: substitutions, timeouts, quarters

**Default:** Include scoring, playmaking, defensive, and negative. Exclude administrative spam.

**Future (`/trackconfig`):** Let users toggle classes per chat.

### Throttling & Spam Control

**Phase 1 (simple):** Wire PBP messages through `ThrottleManager` with priority tiers.
- Scoring plays: normal priority (sent immediately)
- Misses/turnovers: low priority (delayed/suppressed)

**Phase 2 (better):** Batched digest every 20–30s
- Groups all roster matches per cycle
- Single message: "Q2 7:42 — HEZONJA: 2pt Made | LESSORT: 2pt Miss"
- Respects throttle limits while preventing spam

### Per-Player Subscriptions (Defer)

**Not needed yet.** Roster-based model (track all fantasy players) covers the use case. Adding per-player subscriptions adds SQLite schema, UI commands, and subscription mgmt complexity.

**Revisit only if explicit user demand.**

## Key Risks

1. **API reliability.** `live.euroleague.net` is legacy/undocumented. Could break or rate-limit. Graceful degradation already in place.
2. **Chat spam.** 4+ tracked players + misses = 15–25 msg/min. Batching (Phase 2) is the real fix.
3. **Name matching gaps.** PBP uses "LASTNAME, FIRSTNAME" format. Dunkest might differ. `normalizeName()` handles case but not variants (e.g., "De Colo" vs "DE COLO, NANDO").
