# Session: PBP API Benchmark + Payload Analysis

**Date:** 2026-03-13  
**Lead:** Nikola (Data / Integrations)  
**Trigger:** Payload optimization feasibility check for live game polling

## Scope

Measure the cost of a single PBP API call on the PAO vs Zalgiris game (gamecode=305, seasoncode=E2025) to establish baseline metrics for latency, bandwidth, and local processing. Determine whether server-side filtering or reduced-payload strategies are worth implementing.

## Methodology

### 1. API Call Profiling
- Captured raw PBP response: 153.6 KB minified JSON, 578 events
- Measured gzip compression: 10.5 KB on-wire (6.8% ratio)
- Tested 5 network calls: first call ~960ms (cold TLS), subsequent ~50–66ms (warm)

### 2. Local Processing Benchmarks
- JSON.parse only: **0.74ms**
- Full transformation (flatten quarters, carry running scores, map to domain model): **0.78ms**
- Incremental filter (sinceEventId=550, 84 new events): **1.00ms**
- No measurable difference between cold and warm runs on the JS side

### 3. API Behavior Testing (prior session)
- No server-side pagination, filtering, or conditional request support
- `Cache-Control: no-store` confirms anti-cache design
- gzip compression is the only payload reduction that works

## Key Findings

**Conclusion:** Network is 98%+ of total polling cost. At 15s intervals, a single game costs ~2.5 MB/hour. Local CPU work (parsing, filtering) is negligible and not a bottleneck.

**Implication:** Complex payload-reduction strategies (conditional requests, quarter-scoped filters, separate Header-based gating) are not justified. The current implementation (fetch full PBP, client-side sinceEventId filter) is already optimal given the API design.

## Decisions Enabled

1. **Bogdan's low-effort options** are sufficient: skip PBP when rosters not loaded (option #1), reduce poll frequency for cost-conscious deployments (option #2).
2. **No need to implement** ETag-based caching, quarter-scoped fetches, or `/api/Header` gating at this time.
3. **Polling interval is tunable.** If Nikola's live-player recommendation (5s interval) is adopted, the cost scales linearly: 5s → ~5 MB/hour/game. Still acceptable.
